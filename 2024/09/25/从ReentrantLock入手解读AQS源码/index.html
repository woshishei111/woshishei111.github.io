什么是AQS?

AbstractQueuedSynchronizer,中文是抽象队列同步器，一个用来构造锁和同步器的框架，基于它可以简单高效地构造出应用广泛的锁和同步器；

核心思想

锁对象中设置一个 volatile 的标志位 state 来判断锁状态。线程通过 CAS 的方法修改 state 的方法来获取锁，获取锁失败的线程装入一个先进先出的volatile的队列，锁释放之后队列中的线程按照顺序获取锁。

源码阅读

ReentrantLock 是一个可重入的排他锁，初始化的方式有两种，分别为公平锁和非公平锁（是否根据请求锁的顺序来获取锁）。获取/释放锁的方法为 lock() 和 unlock()。

ReentrantLock 的组成元素

组成元素只有一个用于序列化的版本号和一个 Sync 对象。

再看一下 ReentrantLock 的 lock() 和 unlock() 方法

可以看出 Sync 对象用来实现 ReentrantLock 的主要功能

构造方法

ReentrantLock 的两种构造方法

根据命名很容易得出 ReentrantLock 默认初始化为非公平锁，但是也可以选择初始化为公平锁，现在就从代码结构更简单的 FairSync 开始解读 AQS 源码！！！

AbstractOwnableSynchronizer

经过查看 Sync 类的继承关系，看到 AQS(AbstractQueuedSynchronizer) 最早继承了 AbstractOwnableSynchronizer

可以看出，AbstractOwnableSynchronizer 只定义了一个独占线程字段和它的 get/set 方法，它的子类可以通过这两个方法来设置和获取获得了当前锁的线程。

sync.lock()

很有意思的一件事，sync 的 lock 方法调用了 acquire() 方法，参数是 1。继续往下看

acquire 是 sync 的父类 AbstractQueuedSynchronizer 中定义的一个 final，的方法，由此我们可以得出，想要获取锁，就必须调用 AbstractQueuedSynchronizer 的 acquire() 方法，acquire 方法不能被重写。

tryAcquire

tryAcquire 直接抛出异常，所以代码作者希望我们重写 tryAcquire 方法，tryAcquire 可以从名字上看出是尝试获取锁，我们看一下 ReentrantLock 中 FairSync 是怎么实现的
